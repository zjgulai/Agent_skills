The contract

One rule, two halves. Internalize this and the rest follows.

This split has one purpose: portal UI and agent always see the same state. If the agent went around the API to write skill files, the next portal page-refresh would be inconsistent until /api/refresh ran.

The 5 Python lib modules

All five live under agent/lib/ and run inside the portal's venv (no extra dependency tree to maintain).

portal_client.py

HTTP wrapper for the portal REST API. Notable features:

  • ensure_running() — auto-spawns portal/bin/start in background, polls /api/health for up to 20s
  • trust_env=False — bypasses system-level HTTP proxies that would otherwise intercept localhost
  • 180s read timeout — git clone over slow networks can take a minute

domain_inference.py

Keyword scoring against a per-domain dictionary. Mirrors agent/docs/domain-taxonomy.md. Returns uncategorized when no domain scores ≥ 1; agent then prompts user.

index_md_writer.py

The highest-risk module. Six-layer defense:

  • Microsecond-stamped .bak.<ts> backup before every write
  • Anchored regex finds the right H3 domain section
  • Line-level splice preserves exact formatting
  • Atomic write via os.replace — works around macOS APFS leaving dead bytes past EOF on plain write_text
  • Post-write verification: re-parse must still find the change
  • Auto-revert from backup if verification fails

append(name, domain) → remove(name) is byte-identical against the original. Tested.

graph_writer.py

Same defense pattern as index writer, plus auto-creates the %% Domain N - Label block + classDef when adding the first skill in a previously-empty domain. render() spawns render-mermaid.py via playwright + chromium.

doctor.py

5-rule health check. Skill-specific dependency hints (mmdc, .NET, codex CLI, etc.) are declared in a small DEPENDENCY_HINTS table.

Why this shape

Three forces shaped the design:

  • Portal already existed. The portal had been built (FastAPI + Vite/Vue) before the agent. Don't rewrite what works — wrap it.
  • OpenCode auto-discovers .opencode/commands/*.md. So commands are markdown files, not code. The Python lib is what they call.
  • Skill repo lives at ~/.config/opencode/skills/ by OpenCode convention. Don't relocate. Reach in from outside.

What's deliberately not done (v1 boundary)

  • No multi-user / multi-account portal — bound to 127.0.0.1 only
  • No web UI for editing INDEX classification — INDEX edits go through agent commands only
  • No skill version rollback — git pull is the version control
  • No dependency-graph visualization — text-mode /skill-doctor reports cover this
