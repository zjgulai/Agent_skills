§2. Domain Taxonomy

Six domains. One rule: when the classifier is uncertain, ask the user. The classifier never silently miscategorizes.

How inference works

  1. Concatenate name + description, lowercase
  2. For each domain, count how many of its keywords appear (each keyword counted at most once)
  3. Pick the highest-scoring domain. Ties broken by priority order:
ip → desktop → founder → meta → closeout → tooling
  1. If every score is 0 → returns uncategorized; the agent prompts the user to manually pick

The 6 domains

meta AI Engineering Infrastructure

When to enter: the task targets AI agents themselves — repository scaffolding, agent memory, skill libraries, hooks, subagents, plugins, agent self-governance.

Note: bare skill is NOT a meta keyword — too generic; almost every SKILL.md mentions it. Only subagent, agent, and explicit config files are strong signals.

When to enter: code is written, you're at the last mile — commit / PR / release / lint / audit.

Updating the taxonomy

When updating keywords: edit agent/docs/domain-taxonomy.md first, then mirror the change in agent/lib/domain_inference.py KEYWORDS table, finally run pytest regression.

§5. Graph Case Studies

Four graph states, walking through how the graph mutates as new skills are added. The first three are real history (State 0 → 1 → 2); the fourth is a hypothetical exercise.

5.1 State 0: initial 6 skills

The very first state of the project: 6 user-level skills + 5 builtins. One main lifecycle chain founder → meta → desktop → closeout → ip, plus 5 cross-domain reinforcement edges.

  • 6 user skills: ADK, CR, NF, SPT, PAT, SCS (the IP duo)
  • 5 builtins (dashed border): GM, RW, ASR, FE, PW
  • Main chain: SPT → ADK → NF → CR → PAT → SCS (purple → blue → orange → green → teal → teal)
  • No tooling domain block (none yet)

5.2 State 1: after installing guizang-ppt-skill

Triggered by: /skill-install https://github.com/op7418/guizang-ppt-skill

After installing guizang-ppt-skill. The tooling domain block is auto-created and GPS joins as an isolated node. Note: because the tooling domain was empty, the agent automatically added the toolingDomain classDef and the "%% Tooling domain" comment block.

Δ vs State 0: +1 node (GPS in tooling) · +1 classDef (toolingDomain) · +1 self-loop edge · PNG bytes 224 500 → 259 330 (+15.5%)
  • Inference result: tooling domain (matched ppt / swiss style / html, score 3)
  • New node: GPS[guizang-ppt-skill]:::toolingDomain
  • New edge: GPS -. "isolated" .-> GPS (self-loop fallback when same-domain peer is absent)

5.3 State 2: after installing skill-creator

Triggered by: /skill-install https://github.com/anthropics/skills --subdir skills/skill-creator

The current state, after installing skill-creator. The skill's description is too generic ("create new skills"), keyword inference returns uncategorized, the agent prompts the user, and we manually pick meta. The new SC node is in the same domain as ADK, so a "same-domain" dashed edge is added automatically.

Δ vs State 1: +1 node (SC in meta) · +1 same-domain edge to ADK · 0 new classDefs (meta already existed) · PNG bytes 259 330 → 270 529 (+4.3%)
  • Inference result: uncategorized → user manually picks meta
  • New node: SC[skill-creator]:::metaDomain
  • New edge: SC -. "same domain" .-> ADK (meta has an existing peer, so dashed same-domain edge)

5.4 State 3 (hypothetical): installing 3 desktop skills

Imagine: what would the graph look like if you installed 3 new desktop-app skills next (e.g. "tauri-template", "electron-perf-audit", "webview-survival-kit")? The graph would evolve as below — desktop domain expands from 1 node to 4 nodes, with 3 same-domain edges automatically added. Note that NF retains its original cross-domain edges to PW and FE.

  • All 3 new skills infer to desktop (strong keywords: tauri / electron / webview)
  • New nodes: TT, EPA, WSK (SHORT_ID = first letter of each hyphen segment)
  • New edges: each new node -. "same domain" .-> NF (dashed, linking to the first peer)
  • desktop classDef already exists, no new ones created
